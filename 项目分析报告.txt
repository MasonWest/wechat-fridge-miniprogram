【技术栈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
• 前端框架：微信小程序原生框架（WXML + WXSS + JS）
• 后端服务：Supabase（使用 REST API + Edge Functions）
• 样式方案：WXSS（原生样式）
• 数据存储：Supabase PostgreSQL
• 状态管理：小程序原生数据绑定
• 用户标识：微信 open_id（通过 wx.login + Supabase Edge Function）
• 数据隔离：SQL 层面过滤（在查询中添加 user_id 条件）

【核心功能】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 食材管理
   • 添加食材（名称、数量、过期日期、存放位置）
   • 删除食材（标记为"已吃完"）
   • 查看食材列表（支持列表视图和网格视图）

2. 过期提醒
   • 绿色：正常（超过3天过期）
   • 黄色：临期（3天内过期）
   • 红色：已过期

3. 位置管理
   • 默认位置：冷藏、冷冻
   • 自定义位置：支持添加、编辑、删除
   • 位置使用检查：删除前检查是否有食材使用该位置

4. 搜索和筛选
   • 按名称搜索食材
   • 按存放位置筛选
   • 实时过滤

5. 用户管理
   • 微信登录获取 open_id
   • 基于 open_id 的数据隔离
   • 支持复制用户ID

6. 配置管理
   • Supabase 连接配置
   • 连接测试功能
   • 本地存储配置信息

【数据流程】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 应用启动
   • app.js → checkSupabaseConfig()
   • 检查本地存储是否有 Supabase 配置
   • 如果没有，显示配置提示

2. 页面加载
   • index.js → onLoad()
   • 检查本地是否有 open_id
   • 如果没有，调用 loginWechat()
   • 调用 wx.login() 获取 code
   • 调用 Supabase Edge Function 换取 open_id
   • 存储 open_id 到本地
   • 加载食材列表（自动添加 user_id 过滤）
   • 加载自定义位置列表（自动添加 user_id 过滤）

3. 添加食材
   • 用户填写表单
   • 表单验证
   • 调用 addItem({..., user_id}) → supabaseRequest('POST')
   • 数据包含 user_id 字段
   • 重新加载列表（自动过滤当前用户数据）

4. 删除食材
   • 用户点击"已吃完"
   • 确认对话框
   • 调用 deleteItem(id) → supabaseRequest('DELETE')
   • 删除查询同时包含 id 和 user_id 过滤，确保只能删除自己的数据
   • 重新加载列表

5. 添加位置
   • 用户在弹窗中输入位置名称
   • 调用 addLocation(name, userId) → supabaseRequest('POST')
   • 数据包含 user_id 字段
   • 重新加载位置列表

6. 搜索和筛选
   • 用户输入搜索词或选择位置
   • 前端过滤（filterItems()）
   - 实时更新显示

【关键逻辑】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 微信登录流程
   • wx.login() 获取临时 code
   • 调用 Supabase Edge Function
   • Edge Function 调用微信 API
   • 返回 open_id 和 session_key
   • 存储 open_id 到本地
   • 增强的错误日志输出便于调试

2. 数据隔离机制（SQL 层面过滤）
   • 所有查询都添加 user_id=eq.${userId} 条件
   • 所有写操作都包含 user_id 字段
   • 所有删除和更新操作都同时匹配 id 和 user_id
   • 数据库层面允许查询，代码层面保证隔离
   - 性能更好，无额外开销
   - 易于调试，日志清晰
   - 不依赖 Supabase 特殊功能

3. 过期状态判断
   • getDaysUntilExpiry(): 计算距离过期天数
   • getItemStatus(): 根据天数返回状态
     - 天数 < 0: expired（已过期）
     - 天数 ≤ 3: warning（临期）
     - 天数 > 3: normal（正常）

4. 位置管理
   • 默认位置：冷藏、冷冻（不可删除）
   • 自定义位置：可添加、编辑、删除
   • 删除前检查使用情况

5. 视图切换
   • 列表视图：完整信息展示
   • 网格视图：紧凑卡片展示
   - 使用 viewMode 状态控制

6. 配置管理
   • 优先使用本地存储配置
   • 如果没有，使用代码中的默认值
   - 支持测试连接功能

查询示例：
```sql
-- 获取当前用户的物品
SELECT * FROM pantry_items WHERE user_id = '用户_open_id' ORDER BY expiry_date;

-- 添加物品
INSERT INTO pantry_items (name, quantity, expiry_date, location, user_id)
VALUES ('苹果', 5, '2026-04-15', '冷藏', '用户_open_id');

-- 删除物品（只能删除自己的）
DELETE FROM pantry_items WHERE id = 123 AND user_id = '用户_open_id';
```

优点：
• 简单可靠，不依赖 RLS
• 性能更好，单次请求完成
• 易于调试，日志清晰
• 无状态，不依赖连接池
• 易于维护，代码清晰

• 需要确保所有写操作都包含 user_id 字段

【安全考虑】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 已实现：
1. 数据隔离（SQL 层面）
   • 使用微信 open_id 作为用户标识
   • 所有数据表都包含 user_id 字段
   • 所有查询都添加 user_id 过滤条件
   • 所有写操作都包含 user_id 字段
   • 所有删除和更新操作都同时匹配 id 和 user_id
   • 封装在 utils/supabase.js 中，自动处理

2. 用户身份验证
   • 集成微信登录
   • 使用 open_id 作为唯一标识
   • open_id 不会因换设备或清缓存而改变
   • 增强的错误日志输出便于调试

3. 安全的删除和更新
   - 删除和更新操作同时匹配 id 和 user_id
   - 确保用户只能操作自己的数据
   - 即使数据库策略宽松，代码层面也保证安全

⚠️ 仍需注意：
1. Supabase Anon Key 暴露在代码中（默认配置）
   • 建议：移除默认配置，强制用户配置
   - 或者：使用环境变量（小程序有限制）

2. 请求无加密
   - 使用 HTTPS 但数据明文传输
   - 敏感数据建议加密

3. 数据隔离依赖代码
   - 虽然封装良好，但仍需确保所有操作都通过封装的函数
   - 不要绕过 utils/supabase.js 直接调用 API

```sql
DROP POLICY IF EXISTS "Allow all access to pantry_items" ON pantry_items;
DROP POLICY IF EXISTS "Allow all access to custom_locations" ON custom_locations;
```

问题3：添加数据时报错 "new row violates row-level security policy"
原因：get_request_user_id() 函数无法从请求头获取 X-User-ID
测试结果：SELECT get_request_user_id() 返回 NULL
根本原因：Supabase RLS 无法从自定义请求头获取值，RLS 是为 Supabase Auth 设计的

第三阶段（方案重构 - SQL 过滤）：
问题分析：
- RLS 方案在理论上可行，但实际不可靠
- Supabase RLS 无法从自定义请求头获取值
- 微信 open_id 是自定义认证，与 Supabase Auth 不兼容
- 即使使用 RPC 函数设置全局变量，在连接池环境下也不稳定

解决方案：
- 放弃 RLS 方案
- 改用 SQL 层面过滤
- 在所有查询中添加 user_id=eq.${userId} 条件
- 在所有写操作中包含 user_id 字段
- 在所有删除和更新操作中同时匹配 id 和 user_id

   - 添加 getCurrentUserId() 函数
